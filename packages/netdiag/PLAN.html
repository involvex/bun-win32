Goal: Build a first-class, zero-native-dependency, AI-first package @bun-win32/netdiag that turns the existing @bun-win32/{iphlpapi, wlanapi, dnsapi, ws2_32} bindings into an ergonomic, locale-proof, syscall-grade network-diagnostics surface: routing table, socket→PID(+module) map, no-admin ICMP ping + traceroute, typed ARP/neighbor table, adapter census, DNS record queries, live per-interface throughput, and WiFi scan/connect — all returned as TypeScript objects decoded from binary structs, never scraped from netsh/ping.exe/wmic/arp text. Publish it plus an unscoped bun-netdiag alias, with real tests at every step.

Architecture: A non-binding "engine" package (precedent: @bun-win32/terminal and @bun-win32/gpu) of flat root modules over the existing binding packages. There is no marshaling layer and no native build step: each public function does a sizing call, allocates a reused buffer, invokes the bound export, and decodes the returned struct with DataView. The binding packages already exist and already have live, on-hardware proof demos (Appendix A) — this package is the wrapper layer that productizes them. No npm dependencies outside the workspace.

Tech stack: Bun (bun:ffi), TypeScript strict, Biome formatter, the Win32 IP Helper API (iphlpapi.dll), Native WiFi (wlanapi.dll), the DNS API (dnsapi.dll), and Winsock (ws2_32.dll) — all OS-shipped DLLs already in System32 on every Windows machine. Windows-only, Bun-only — stated proudly, not apologetically.

⚡ The performance doctrine (this is what makes netdiag worth shipping — read before every hot-path phase)

The entire pitch is "every sample is a syscall, not a process spawn" — a live dashboard polling at sub-millisecond cadence, impossible when default-gateway/ping/node-wifi/systeminformation each cost a CreateProcess + localized text parse (tens of milliseconds and locale-dependent breakage; systeminformation even self-throttles network stats to a 500 ms floor). A diagnostics engine that re-allocates and re-parses on every poll throws that advantage away. Performance is a standing, always-on audit, not a closing benchmark — every hot-path phase ends by analyzing its inner loop down to allocations, the sizing-call round-trip, the struct-decode loop, and FFI-boundary crossings. Treat every number below as a floor, not a finish line — keep hunting for wins this plan never anticipated (the generative pass, G.3). Rules (re-measured per host in S0.6b — Bun runs on JavaScriptCore, not V8; treat V8 folklore as suspect until measured; full data in Appendix G):

  • Reuse, never re-allocate, in any per-poll loop. A Buffer.alloc per sample measured ~1064 ns (and the memset dominates); a fresh DataView per parse is +20 ns; a per-row toArrayBuffer+DataView is ~29× a single reused view. The sizing-call pattern (call with NULL → get required bytes → allocate → call again) is the engine's signature shape: keep one growable data buffer and one 4-byte size buffer per table source, grow geometrically, and reuse them across polls. Decode by wrapping the returned region in ONE DataView and reading typed accessors at computed offsets; pre-size result arrays by dwNumEntries.
  • Read buffer.ptr inline at the FFI call — do not cache it across allocations. Ground truth (repo memory): small Buffer backing stores relocate after later allocations, so a cached .ptr can dangle → E_INVALIDARG/garbage. This overrides BUN_GPU's "cache .ptr once" lever for this package's allocation-heavy sizing-call paths. The getter itself costs ~7.3 ns (it re-runs ptr(this)); hoist it only within a tight inner loop that does NOT allocate between uses, and re-read it after any allocation.
  • Avoid bigint in the hot path; minimize FFI crossings. The FFI floor is ~5 ns/call; a u64→bigint handle return adds ~10 ns; a fresh BigInt() arg per call is ~16 ns (hoist constant handle args out of poll loops). The socket→PID join key is a number (PID), never a bigint. The macro-opt is structural: one GetExtendedTcpTable call returns the whole table — never one syscall per connection.
  • Use DataView typed accessors, not hand-rolled byte math (on JSC manual byte-OR is ~1.13× slower — the opposite of V8 lore); Big(U)int64Array views are the slowest u64 path and often unaligned-illegal. IPv4 address bytes, ports (network byte order — one byteswap), and MIB counters are all DataView.getUint32/getUint16 reads. JSC shows no monomorphism penalty — don't contort for it.
  • Pace dashboards correctly. Bun.sleep (async) quantizes to ~15.6 ms → a 64 fps ceiling and stutter; Bun.sleepSync is NOT quantized (~1.5 ms), and the proven sub-frame path is a high-resolution waitable timer + setImmediate yield (repo memory: Windows 60fps frame pacing). The "sub-millisecond sampling" claim is about the syscall cost, not the wall-clock poll interval — measure and state both honestly.
  • WiFi scan is intentionally slow and that is correct. Two paths: the OS-cached available-network list (instant — what the existing demos use) and a FRESH scan, WlanScan → ~4 s settle → WlanGetAvailableNetworkList (MSDN-sanctioned; verified the repo's wlanapi demos never call WlanScan, so netdiag is the first to implement it — and WlanScan may FLUSH the prior list, so you must wait the full ~4 s before reading). Do not "optimize" with WlanRegisterNotification (foreign-thread callback — crash class; MSDN itself blesses "time out and query the current interface state instead of waiting for a notification"). Expose a triggerScan flag so callers pick cached (instant) vs fresh (~4 s); document the latency; it is not a hot path.
  • Every hot path gets an isolated microbenchmark (Bun.nanoseconds(), 200k+ warm-up, dead-code sink, median ns/op — the terminal.bench.ts convention): a full route-table poll, a socket→PID table poll, a single IcmpSendEcho, an ARP-table decode, the GetIfTable2 throughput delta. Benchmarks DRIVE optimization, not just the README table. A perf baseline is recorded in Phase 0 and gated at release (S14.6) exactly like a tsc error.
  • Prove it, don't hope — Bun ships bun:jsc. Every decode loop can be verified optimized: numberOfDFGCompiles(fn) > 0 (timing optimized code, not the interpreter), profile(cb) tier breakdown (assert FTL/DFG dominates the table-decode), reoptimizationRetryCount(fn) (climbing = type-unstable deopt), jscDescribe(obj) (Structure/IndexingType X-ray of the row objects), heapStats().objectTypeCounts deltas (gate per-poll allocation at zero after warmup). Full verified toolkit + measured JSC levers in Appendix G.4/G.5 — use them in every microbench and audit.
  • Don't satisfice — beating the baseline is the job. Once a path is correct, profile() it to find the ACTUAL dominant cost (often something this plan never listed — the sizing-call double-invoke, a redundant per-row copy, a bigint conversion), form a hypothesis, microbench an alternative, keep it only if it wins AND stays correct, cast-free, and readable — then go again until returns genuinely diminish. Can a read be hoisted, a table cached for the poll interval, two syscalls fused, a result returned as a typed-array view instead of an object array? Any lever you discover that this plan didn't list — prove it with the G.4 tools and add it to Appendix G.5. Never stop at "fast enough."

S2.1Create packages/netdiag/win32.ts NEW — the sizing-call helper. Nearly every IP Helper table API uses the two-call idiom: call with a NULL buffer to learn the byte count (returns ERROR_INSUFFICIENT_BUFFER 122 or ERROR_BUFFER_OVERFLOW 111), allocate, call again. This is the engine's hottest shape and the place the perf doctrine lives. Reference implementation — type-check, run, fix: 
import type { Pointer } from 'bun:ffi';

const ERROR_SUCCESS = 0;
const ERROR_INSUFFICIENT_BUFFER = 122;
const ERROR_BUFFER_OVERFLOW = 111;

/**
 * Run a Win32 sizing-call pair. `invoke(bufferPtr, sizePtr)` must accept a NULL
 * data pointer for the sizing pass and return a Win32 error code. Reuses the
 * caller-provided growable buffers across polls — never allocate per sample.
 * .ptr is read INLINE here (Buffers relocate after allocation — repo ground truth).
 */
export function sizedCall(state: SizedBufferState, invoke: (dataPtr: Pointer | null, sizePtr: Pointer) => number): Buffer {
  state.sizeView.writeUInt32LE(state.buffer.byteLength, 0);
  let error = invoke(state.buffer.ptr, state.sizeView.ptr);
  if (error === ERROR_INSUFFICIENT_BUFFER || error === ERROR_BUFFER_OVERFLOW) {
    const required = state.sizeView.readUInt32LE(0);
    if (required > state.buffer.byteLength) state.buffer = Buffer.allocUnsafe(required); // grow + reuse
    state.sizeView.writeUInt32LE(state.buffer.byteLength, 0);
    error = invoke(state.buffer.ptr, state.sizeView.ptr); // re-read .ptr: buffer just moved
  }
  if (error !== ERROR_SUCCESS) throw new Win32Error(error);
  return state.buffer;
}

export class SizedBufferState {
  buffer: Buffer;
  readonly sizeView: Buffer;
  constructor(initialBytes = 0x0000_4000) {
    this.buffer = Buffer.allocUnsafe(initialBytes);
    this.sizeView = Buffer.allocUnsafe(4);
  }
}

export class Win32Error extends Error {
  readonly code: number;
  constructor(code: number) {
    super(`Win32 error ${code}: ${win32ErrorMessage(code)}`);
    this.code = code;
    this.name = 'Win32Error';
  }
}

Only we do it without the node-gyp/shell tax (incumbents exist but pay a price netdiag doesn't):

G.3 — The discipline

  1. Baseline (S0.6b) records the table-poll/ICMP/codec floors on the release host before any work.
  2. Every table-source phase (3, 4, 5, 9) ends with a microbenchmark asserting zero per-poll allocation after warmup and (for self-alloc APIs) no leak across 100k polls.
  3. Release gate (S14.6) re-runs everything; a >~10% regression vs baseline OR any forbidden hot-loop pattern blocks release.
  4. README numbers (samples/sec, ns/poll, the spawn-vs-syscall multiple) come only from the gated run — never copied from this appendix.
  5. Generative pass (the real spirit — every hot-path phase). profile() the path to find the true hot spot, then try at least one optimization this plan didn't anticipate — a table cached for the poll interval, a fused two-syscall read, a result returned as a typed-array view instead of an object array, a cheaper sizing-call seed size — measure it, fold any proven win into G.5. "Meets baseline" is the floor; beating it is the job.
  6. Methodology (terminal.bench.ts convention): Bun.nanoseconds(); warm ≥200k iters; assert numberOfDFGCompiles(fn) > 0 (else you're timing the interpreter); folded sink vs dead-code elimination; vary offsets to defeat hoisting; gcAndSweep() between (not within) trials; median of ≥5.
